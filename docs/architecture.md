# Architecture

## System Overview

Kaskad Oracle is a **multi-source price oracle** that runs inside a Trusted Execution Environment. It replaces centralized oracle dependencies (Chainlink, etc.) with a self-hosted, cryptographically attested price feed for the Kaskad lending protocol on Galleon testnet.

### Core Principle

> The oracle binary runs inside an enclave that has **no network, no disk, no SSH**. All external communication happens through a controlled VSOCK channel. The signing key is generated **inside** the enclave and never leaves it. An attestation document proves to anyone that the key was generated by this specific code running in this specific enclave.

## Aggregation Strategy

### Why Weighted Median (not Mean)

The mean is heavily influenced by outliers. A single bad feed can swing the price. The **median** is robust — even if 2 out of 5 sources report garbage, the result is still correct as long as the majority agrees.

We use **volume weighting** because exchanges with more trading volume have more reliable price discovery. Binance ETHUSDT with $2B daily volume should count more than CoinGecko's aggregated estimate.

### Outlier Rejection Pipeline

```
Input:  [$2129.27, $2129.26, $2129.05, $2129.14, $2128.37]

Step 1: Sort → [$2128.37, $2129.05, $2129.14, $2129.26, $2129.27]

Step 2: Median = $2129.14

Step 3: Absolute deviations from median:
        [0.77, 0.09, 0.00, 0.12, 0.13]

Step 4: MAD = median of deviations = 0.12

Step 5: Threshold = 3.0 × 1.4826 × 0.12 = 0.534

Step 6: Reject any |price - median| > 0.534
        $2128.37: |2128.37 - 2129.14| = 0.77 > 0.534 → REJECTED
        Others: all within threshold → KEPT

Output: [$2129.05, $2129.14, $2129.26, $2129.27]  (4 sources)
```

The `1.4826` constant is the consistency factor that makes MAD comparable to standard deviation for normally distributed data.

### Per-Asset Thresholds

Different assets need different sensitivity. We use the same model as Chainlink:

| Asset | Deviation trigger | Heartbeat | Rationale |
|-------|-------------------|-----------|-----------|
| ETH/USD | 50 bps (0.5%) | 1 hour | Liquid, moderate volatility |
| BTC/USD | 50 bps | 1 hour | Same |
| KAS/USD | 200 bps (2.0%) | 30 min | Exotic, higher volatility |
| USDC/USD | 10 bps (0.1%) | 24 hours | Stablecoin, trigger on depeg |
| IGRA/USD | 0 (always) | 24 hours | Governance-set, push on change |

## Signature Scheme

The oracle signs price updates using EIP-191 personal signatures that are verifiable on-chain via `ecrecover`:

```
Payload = abi.encodePacked(
    bytes32 assetId,        // keccak256("ETH/USD")
    uint256 price,          // 212926000000 (8 decimals)
    uint64  timestamp,      // unix seconds
    uint8   numSources,     // number of sources used
    bytes32 sourcesHash     // keccak256(source_data)
)

messageHash = keccak256(Payload)

ethHash = keccak256("\x19Ethereum Signed Message:\n32" || messageHash)

(v, r, s) = ecdsaSign(ethHash, privateKey)
```

On-chain, the contract recovers the signer address via `ecrecover(ethHash, v, r, s)` and checks it matches the registered enclave address.

## TEE Trust Model

```
┌──────────────────────────────────────────────────────────────┐
│                    Trust Chain                                │
│                                                               │
│  1. Code is open-source → reproducible build                  │
│  2. Docker image hash → PCR0 in attestation document          │
│  3. Attestation signed by AWS/Intel root key                  │
│  4. On-chain contract verifies attestation once                │
│  5. Contract stores enclave's public key                      │
│  6. All subsequent price updates verified via ecrecover       │
│                                                               │
│  Anyone can:                                                 │
│  ✓ Build the same Docker image, get the same PCR0 hash       │
│  ✓ Verify that PCR0 matches what's registered on-chain       │
│  ✓ Be certain the signing key was generated by this code     │
│  ✓ Verify every price update's signature                     │
│                                                               │
│  Nobody can (not even the operator):                         │
│  ✗ Extract the signing key from the enclave                  │
│  ✗ Modify the code without changing PCR0                     │
│  ✗ Submit prices that weren't computed by the oracle          │
│  ✗ Access the enclave's memory from the host OS              │
└──────────────────────────────────────────────────────────────┘
```

### Local Development vs Production

| | Local (MockSigner) | Production (EnclaveSigner) |
|---|---|---|
| Key storage | env var / random | Generated inside enclave, never exported |
| Key binding | None | Bound to enclave measurement (PCR0) |
| Attestation | None | AWS Nitro (CBOR/COSE) or Intel TDX (DCAP) |
| Network | Direct HTTP | VSOCK proxy only |
| Tamper proof | No | Hardware-enforced memory encryption |

The `OracleSigner` trait makes the switch transparent — same oracle logic, different key management backend.

## Technology Stack

| Layer | Technology | Why |
|-------|-----------|-----|
| Oracle binary | Rust | Memory safety, no_std/WASM path, alloy/revm ecosystem |
| HTTP client | reqwest | Battle-tested, async, TLS |
| EVM types | alloy-primitives, alloy-sol-types | no_std compatible, Paradigm-maintained |
| Signing | k256 (secp256k1) | Pure Rust, audited, no OpenSSL dependency |
| Async runtime | tokio | Standard for Rust async |
| Smart contracts | Solidity + Foundry | Standard EVM tooling |
| TEE | AWS Nitro Enclaves | Available, documented, operator has experience |
| Future: ZKP | circom + snarkjs (or ark-groth16) | Prove aggregation correctness |
| Future: WASM | wasm32-wasi | Minimize TCB inside enclave |
